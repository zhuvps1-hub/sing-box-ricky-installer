# iWAN Gateway v7 重构路线图

## 1. 当前故障根因

报错：

```text
inbounds[0].address: json: unknown field "address"
```

说明面板把通用 TUN 字段 `address` 写进了定制 iWAN inbound。当前 iWAN 核心要求地址池使用 `address_pool`。旧面板为了兼容历史配置，同时读取和提交两个字段，导致不支持的字段进入最终配置。

本轮修复采用三层防护：

1. 浏览器或旧配置提交的 `address` 只作为迁移别名；
2. 后端统一转换为 `address_pool`，未知字段不会进入 sing-box；
3. 读取旧配置时先做内存迁移，下一次保存会自动清除旧字段。

## 2. 已有服务器立即修复

先预览：

```bash
python3 tools/repair_iwan_schema.py
```

确认检测到旧字段后执行：

```bash
sudo python3 tools/repair_iwan_schema.py --apply
```

脚本会依次完成：

```text
读取配置
→ 只迁移 iWAN inbound
→ 生成候选文件
→ sing-box check
→ 创建时间戳备份
→ 原子替换
→ 重启并检查服务
→ 失败自动恢复备份
```

不要直接使用 `sed` 修改 JSON，也不要在没有备份的情况下覆盖 `/etc/sing-box/config.json`。

## 3. 目标架构

```text
Web UI
  │  只提交稳定 API 模型
  ▼
HTTP / Auth / CSRF
  │
  ▼
Application Service
  ├─ Config transaction
  ├─ Node management
  ├─ Routing management
  ├─ iWAN management
  └─ mosdns management
  │
  ▼
Compatibility Adapters
  ├─ Ricky-Hao sing-box adapter
  ├─ Schema migration
  └─ Version capability detection
  │
  ▼
System Services
  ├─ sing-box
  ├─ mosdns
  └─ systemd
```

核心原则：Web 表单字段不能直接复制到 sing-box JSON。所有数据必须先进入稳定的内部模型，再由版本适配器生成目标配置。

## 4. 分阶段重构

### P0：故障止血

- 统一 iWAN 地址池字段为 `address_pool`；
- 自动迁移旧 `address`；
- 增加端口、MTU、CIDR 校验；
- 加入回归测试和修复工具；
- 发布 v7.0.1。

### P1：配置事务层

把当前散落在 HTTP Handler 中的写入逻辑拆为：

```text
load → normalize → validate → render → core check
→ backup → atomic replace → restart → health check → commit/rollback
```

建议目录：

```text
panel/gateway/
├── api/
├── application/
├── domain/
│   ├── iwan.py
│   ├── node.py
│   └── routing.py
├── adapters/
│   ├── singbox_ricky.py
│   ├── mosdns.py
│   └── systemd.py
├── infrastructure/
│   ├── atomic.py
│   ├── backup.py
│   └── locks.py
└── security/
```

### P2：解除版本叠层

当前 `v62 → v63 → ... → v69` 的模块叠层难以维护。后续应改为：

- 一个主程序包；
- 明确的语义版本；
- 数据迁移脚本；
- 每个版本只保留发布标签，不再通过多层 Python import 继承旧版本。

### P3：权限隔离

面板目前需要修改系统配置和重启服务，不应长期以完整 root 权限运行。建议拆成：

```text
非特权 Web 进程
      │ Unix socket
      ▼
最小权限 helper
```

helper 只允许固定操作：

- 校验候选配置；
- 写入指定配置文件；
- 重启白名单服务；
- 创建和恢复指定目录下的备份。

同时加强 systemd：

```text
NoNewPrivileges=true
PrivateTmp=true
ProtectHome=true
ProtectSystem=strict
PrivateDevices=true
RestrictSUIDSGID=true
LockPersonality=true
RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6
```

面板默认监听 `127.0.0.1`，公网访问通过 Caddy/Nginx HTTPS 或 WireGuard 管理网完成。

### P4：发布系统

稳定发布流程应为：

```text
PR 测试
→ 合并
→ 生成不可变 release commit
→ 计算 Git Blob/SHA256
→ 签名 release manifest
→ 全新机安装测试
→ 升级测试
→ 回滚测试
→ 发布
```

禁止发布清单指向持续变化的 `main`。安装入口可以在 `main`，但入口必须下载固定版本的已签名清单。

## 5. 测试矩阵

每次发布至少验证：

- 全新 Debian 12 / Ubuntu 24.04；
- 已有无 iWAN 配置；
- 已有 `address_pool` 配置；
- 已有旧 `address` 配置；
- 面板端口冲突；
- sing-box check 失败；
- sing-box restart 失败；
- 网络中断后自动保存恢复；
- mosdns 配置失败回滚；
- 升级失败恢复旧面板。

## 6. 发布前安全检查

- 面板端口不得默认向全网开放；
- Cookie 在 HTTPS 下增加 `Secure`；
- 登录限速状态持久化或接入反向代理限速；
- 备份目录权限固定为 `0700`，配置文件为 `0600`；
- 日志接口过滤密码、节点链接和认证头；
- 所有子进程调用使用参数数组，不使用 shell 拼接；
- 发布文件、二进制和清单均有不可变校验。
